## Overview

Vantik ships as a single Docker Compose stack in the main repository. It runs
the webapp, the API server, and all backing services (PostgreSQL, Redis,
SuperTokens, Typesense), and the server applies database migrations on
startup — no separate deployment repo or setup script is needed.

This guide gets you a working deployment quickly. Scaling, backups, and
hardening beyond the basics are up to you.

## Requirements

- Docker, or Podman with the compose provider (a Podman VM needs at least
  8 GB of memory to build the images)
- 4 CPU / 8 GB RAM is a comfortable baseline

## Instructions

1. Clone the repository and create your `.env`:

```bash
git clone https://github.com/cmunte132/vantik.git
cd vantik
cp .env.example .env
```

2. For anything beyond a localhost test, edit `.env`:

   - Set `FRONTEND_HOST` and `BACKEND_HOST` to your domain
   - Change `POSTGRES_PASSWORD`, `TYPESENSE_API_KEY`, and `TRIGGER_TOKEN`

3. Start the stack:

```bash
docker compose up -d
```

4. Open the webapp (http://localhost:3000 by default, or your
   `FRONTEND_HOST`) and sign in with your email address. Without SMTP
   configured, the magic login link is printed to the server log:

```bash
docker compose logs server | grep -A5 "magic link"
```

## Optional services

Everything below is optional. The server logs an error and continues if a
service is not configured.

### Email (SMTP)

Magic-link logins and notification emails are sent via SMTP when configured;
otherwise login links land in the server log. Any provider that supports SMTP
authentication works (Gmail, Amazon SES, SendGrid, ...):

```bash
SMTP_HOST=your-smtp-host
SMTP_PORT=587
SMTP_USER=your-smtp-username
SMTP_PASSWORD=your-smtp-password
SMTP_DEFAULT_FROM=noreply@yourdomain.com
```

### AI features

AI features (title generation, description elaboration, breaking issues into
sub-issues, filter suggestions) use the first configured provider:

```bash
# Hosted providers (either one)
OPENAI_API_KEY=your-openai-key
ANTHROPIC_API_KEY=your-anthropic-key

# Or a local model via ollama
LOCAL_MODEL=gemma2:2b
OLLAMA_HOST=http://your-ollama-host:11434
```

Semantic search re-ranking can additionally use Cohere
(`COHERE_API_KEY`); without it, Typesense's built-in ranking is used. The
models default to `embed-english-v3.0` and `rerank-english-v3.0` and can be
overridden with `COHERE_EMBED_MODEL` and `COHERE_RERANK_MODEL`.

Changing the embedding model invalidates the existing index. To rebuild it,
delete the Typesense `issues` collection and restart the server — on boot the
server recreates the collection and re-indexes every workspace from Postgres:

```bash
curl -X DELETE \
  -H "X-TYPESENSE-API-KEY: $TYPESENSE_API_KEY" \
  http://localhost:8108/collections/issues
```

The same rebuild happens automatically when a server upgrade adds fields to the
search schema — the collection is detected as stale, dropped and refilled.

### File attachments

Issue attachments are stored in Google Cloud Storage or Amazon S3. Without a
configured provider, attachment uploads are unavailable — everything else
works.

```bash
# Google Cloud Storage
STORAGE_PROVIDER=GCP
BUCKET_NAME=your-bucket-name
GCP_SERVICE_ACCOUNT_FILE=/path/to/service-account.json  # Storage Object Admin role
PUBLIC_ATTACHMENT_URL=https://your-domain.com/api

# Amazon S3
STORAGE_PROVIDER=AWS
BUCKET_NAME=your-bucket-name
AWS_REGION=your-region
AWS_ACCESS_KEY_ID=your-access-key
AWS_SECRET_ACCESS_KEY=your-secret-key
PUBLIC_ATTACHMENT_URL=https://your-domain.com/api
```

Your bucket needs a CORS configuration allowing your frontend origin:

```json
[
  {
    "AllowedHeaders": ["*"],
    "AllowedMethods": ["GET", "PUT", "POST", "DELETE", "HEAD"],
    "AllowedOrigins": ["https://your-domain.com"],
    "ExposeHeaders": ["ETag"],
    "MaxAgeSeconds": 3000
  }
]
```

### Background jobs (trigger.dev)

[Actions](/docs/actions/overview) — the automation framework — and some
background jobs (user notifications, automated cycle transitions, scheduled
tasks) run on [trigger.dev](https://trigger.dev). Core issue tracking works
without it.

To enable, deploy trigger.dev yourself
([self-hosting guide](https://trigger.dev/docs/self-hosting)) and point
Vantik at it in `.env`:

```bash
TRIGGER_API_URL=https://your-trigger-instance
TRIGGER_DATABASE_URL=postgresql://user:pass@host/trigger
```

:::note
The automation subsystem is being reworked in this fork (see the
[roadmap](https://github.com/cmunte132/vantik#roadmap-planned-direction-not-yet-built)),
so treat trigger.dev integration as legacy-supported rather than the
long-term direction.
:::

## FAQ

**Q: Is there a hosted cloud version?**
A: No. Vantik is self-hosted only.

**Q: Can I run Vantik without trigger.dev?**
A: Yes — all core issue tracking works. You lose Actions, user notifications,
and automated cycle transitions.

**Q: Which storage providers are supported for attachments?**
A: Google Cloud Storage and Amazon S3.

If there are any questions that we couldn't answer here, feel free to open an
issue on [GitHub](https://github.com/cmunte132/vantik/issues).
